# 国际化词典与审核闭环规格

## 1. 目标

建立一套可在本地独立运行的国际化词典流程。词条由本地 TMS 审核，应用在启动和构建前生成自己的运行时词典；不依赖数据库、远程发布接口或第三方词典工具。

本规格以 `apps/poker-web` 为首个接入应用，但所有机制应可供未来的 `apps/*` 应用复用。

## 2. 术语

| 术语 | 含义 |
| --- | --- |
| 权威词典 | 已审核通过、可作为翻译来源的完整词典。 |
| 运行时词典 | 应用启动或构建时由权威词典生成的副本。 |
| 待审核任务 | 新词条经过 Agent 生成候选译文后、尚未人工批准的记录。 |
| 编码 | 形如 `P000001` 的消息标识符。 |
| 词条 | 一个编码及其中文、英文译文。 |

## 3. 目录与 Git 规则

```text
i18n-data/
  web/
    en.json                 # 权威英文词典，应提交
    zh-CN.json              # 权威中文词典，应提交
    pending-jobs.json       # 待审核任务与编码保留记录，应持久化

apps/poker-web/
  src/
    locales/
      en.json               # 运行时生成，Git 忽略
      zh-CN.json            # 运行时生成，Git 忽略
```

`packages/i18n` 不保存任何应用业务词典，只保存通用的语言类型、翻译 API、校验能力和路由中间件能力。

`i18n-data/web` 是 Web 词典的唯一权威来源。`apps/poker-web/src/locales` 可以被删除并重新生成，不能作为审核或发布的来源。

## 4. 职责边界

### 4.1 TMS UI

- 提供单条和批量中文词条录入。
- 展示精确匹配结果与相似词条建议。
- 展示 Agent 生成的英文候选译文。
- 提供逐条编辑、批准、拒绝或取消待审核任务的界面。

### 4.2 TMS API

- 仅作为本地审核工作流后端，继续限制为回环地址访问。
- 负责查重、自动分配与保留编码、创建任务、调用 Agent、保存审核状态和原子写入权威词典。
- 不作为 Web 的词典下载接口；Web 不通过 HTTP 调用 TMS。

### 4.3 packages/agents

- 仅基于“系统分配的编码 + 中文原文”生成英文候选。
- 不直接修改任何词典、任务状态或文件。

### 4.4 应用

- 代码仅引用消息编码，例如 `t("P000085")`。
- 启动和生产构建前执行本地词典同步与校验。
- 不负责分配编码、审核译文或维护权威词典。

## 5. 词条录入与审核流程

### 5.1 输入与查重

1. 用户在 TMS 输入一条或多条中文词条。
2. 系统对每条词条执行规范化后的中文精确匹配。
3. 系统同时返回相似词条建议，用于人工避免语义重复。
4. 精确匹配到的词条直接返回已有 `Pxxxxxx`；不得新建任务，也不得调用 Agent。
5. 未匹配到的词条作为新增候选进入下一步。

### 5.2 自动分配编码

1. TMS 服务端为每个新增候选自动分配一个未使用的 `Pxxxxxx`。
2. 编码必须满足 `P` 加六位数字，且不得为 `P000000`。
3. 分配时必须同时检查：
   - 已批准的中英文词典；
   - 所有未完成、待审核、处理中任务；
   - 同一批次内已经分配的编码。
4. 编码分配与保留必须在单一串行或锁定流程中进行，避免并发重复。
5. 编码一经分配即写入持久化的待审核记录；TMS 重启后不得重新分配。

### 5.3 翻译与审核

1. TMS 将新增词条批量交给 `packages/agents`，只请求英文候选。
2. Agent 返回候选后，任务进入待审核表格。
3. 用户可逐条修改候选英文、批准、拒绝或取消。
4. 批准前不得修改权威词典。
5. 批准时，TMS 校验：
   - 编码仍被该任务保留；
   - 中文、英文均为非空字符串；
   - 中英文位置占位符集合一致；
   - 编码在权威词典中尚未存在。
6. 校验通过后，TMS 原子地将批准词条增量合并到 `i18n-data/web/en.json` 与 `i18n-data/web/zh-CN.json`，再完成任务状态迁移。
7. 拒绝或取消时，释放该任务保留的编码；已批准编码不得释放或复用。

## 6. 增量原则

- 每次录入只将未存在的中文词条发送给 Agent。
- 已存在词条直接复用编码，不生成新翻译、不创建审核任务。
- 每次批准只合并本次批准的条目，不修改未涉及的既有词条。
- 为保证崩溃安全，JSON 文件可采用“写入临时文件后原子替换”的方式落盘；这属于文件写入策略，不表示全量重翻或全量审核。
- 运行时词典可为完整快照，因为应用在任意页面都可能查询任意已批准编码。

## 7. 运行时词典生成

### 7.1 生成命令

每个需要国际化的应用在自身 `package.json` 声明 `updateI18nResource`。根目录的 `pnpm run i18n:sync` 只负责发现并执行这些应用命令。

`apps/poker-web` 的命令必须：

1. 读取 `i18n-data/web/en.json` 与 `i18n-data/web/zh-CN.json`；
2. 校验词典结构和双语编码集合；
3. 生成 `apps/poker-web/src/locales/en.json` 与 `apps/poker-web/src/locales/zh-CN.json`；
4. 使用原子替换，避免开发服务器读取到半写入文件。

### 7.2 执行时机

- `predev`：每次本地开发服务启动前执行。
- `prebuild`：每次生产构建前执行。
- 可通过根命令手动执行指定应用的同步。

同步失败必须阻断启动或构建，不得静默回退到过期词典。

## 8. Web 构建校验

Web 在构建前必须扫描 `apps/poker-web/src` 中引用的 `Pxxxxxx` 编码，并验证：

1. 每个被引用编码同时存在于英文与中文权威词典；
2. 两种语言的编码集合一致；
3. 两种语言的占位符集合一致；
4. 运行时词典与本次权威词典同步一致；
5. 不存在未批准的编码被应用引用。

任何一项失败时，构建必须失败并列出缺失或冲突的编码。

## 9. 语言路由

完成词典闭环后，Web 使用 `packages/i18n` 提供的通用路由中间件实现：

- 英文默认无前缀：`/`、`/settings`；
- 中文使用前缀：`/zh-CN`、`/zh-CN/settings`；
- 语言切换必须保留当前 pathname；
- Cookie 仅用于首次访问时的语言偏好或重定向，不得作为唯一语言来源；
- 所有应用导航链接必须保留当前语言前缀。

## 10. 非目标

- 本阶段不引入数据库保存词典或审核任务。
- 本阶段不提供 Web 从 TMS API 拉取词典的接口。
- 本阶段不使用第三方词典生成工具。
- 本阶段不允许 Agent 自动批准或直接写入词典。

## 11. 验收标准

1. 批量输入中，已存在中文词条只返回既有编码，Agent 不被调用。
2. 新增中文词条获得唯一的系统分配编码，并在 TMS 重启后仍保持保留状态。
3. Agent 只能产生英文候选，批准前 `i18n-data/web` 不发生变化。
4. 批准后两个权威词典同时新增对应词条，且占位符校验通过。
5. `pnpm run i18n:sync` 能为 Web 生成 Git 忽略的运行时词典。
6. Web 构建对缺失、未批准或双语不一致的编码失败。
7. 中文切换从 `/settings` 跳转到 `/zh-CN/settings`，英文切换反向跳转；刷新和直达链接均可正常工作。
